# Kyber Rusty 

tl:dr; A **quantum safe** implementation of PASETO v4 (lighter weight alternative to JWT) for OpenResty using Rust libraries with client side encryption operations with WASM.  This is fricken cool! 

A paired Rust + Lua + WebAssembly implementation of:

- ML-KEM-768 (post-quantum key exchange)
- XChaCha20-Poly1305 (authenticated encryption)
- Blake2b-512 (key derivation)
- PASETO v4.local token minting and verification

Designed for:

- **Server-side**: [OpenResty](https://openresty.org/en/) (Lua + native Rust `.so`)
- **Client-side**: Browser (WASM + generated JS)

- I built this as a component of a FOSS project I'm working on.  It's released under the MIT license.  If you want to fork it or whatever.  I'd love to hear about it if anyone actually uses it for something.

## Overview

This project is split into two halves that work together:

### 1. `kyber-rusty` (Server / OpenResty)

- Rust builds a native shared library (`.so`)
- Lua loads it via FFI
- `paseto.lua` implements PASETO v4.local using those primitives

Flow:

Rust (.so)

↓

kyber.lua (FFI bindings)

↓

paseto.lua (token mint/verify)

↓

OpenResty application

### 2. `kyber-wasm` (Client / Browser)

- Rust compiled to WebAssembly using `wasm-pack`
- JS wrapper (`kyber_wasm.js`) generated automatically
- Provides browser-side crypto and token verification

Flow:

Rust (wasm-bindgen)
↓
WASM binary + JS wrapper
↓
Browser application

---

## How the Pieces Fit Together

1. Server generates ML-KEM keypair (via Rust `.so`)
2. Client receives **public key**
3. Client calls:

```js
   encaps(public_key)
```

→ returns:

* ciphertext
* shared_secret

4. Server decapsulates → same shared secret

5. Server mints PASETO token using `paseto.lua`

6. Client verifies token using:

   ```js
   verify_paseto(shared_secret, token)
   ```

Both sides use identical:

* key derivation
* PAE encoding
* encryption scheme

## OpenResty Integration (Minimal)

### 1. Place the shared library

Copy:

```
libkyber_rusty.so
```

to a location accessible by Lua (e.g., `/usr/local/lib` or your project directory).
see genkyberkeys.sh for suggestions if you're not sure where to put this.

---

### 2. Load the Lua module

```lua
local kyber = require("kyber")
local paseto = require("paseto")
```

---

### 3. Generate your keys
You can use genkyberkeys.sh to do this, or you can use whatever other method you want.
I suggest looking at that file for how to do this, because it's pretty self explanitory.
The keys then need to be saved somewhere (like in a .env file) and loaded so that OpenResty can read them.
I assume that if you're using OpenResty in a project, you have an idea of ways to do this.  If I assume wrong, feel free to email me, I'm very friendly.

### 4. Example: Mint a token

```lua
local token, err = paseto.mint(shared_secret, {
    sub = "user_id",
    jti = "unique_token_id"
})
```

---

### 5. Example: Verify a token

```lua
local claims, err = paseto.verify(shared_secret, token)
```

## Security Considerations

This project combines PASETO v4.local with a post-quantum key establishment layer (ML-KEM-768). While the design aims to reduce common implementation risks, correct and secure usage still depends on the surrounding application.

### PASETO Version and Mode

This implementation uses:

- **PASETO v4.local**

This means:

- Tokens are **encrypted and authenticated (symmetric AEAD)**
- Token contents are **not readable without the shared secret**

This is different from `v4.public`, which is **signed only** and does not provide confidentiality.

[PASETO v4 spec](https://github.com/paseto-standard/paseto-spec/blob/master/docs/01-Protocol-Versions/Version4.md).

---

### Claim Validation

Cryptographic verification alone isn't enough.

This implementation validates:

- `exp` (expiration)
- Required claims (`sub`, `jti`)

Applications are responsible for enforcing any additional constraints, such as:

- issuer (`iss`)
- audience (`aud`)
- subject scoping
- replay protection tied to `jti`

Failure to validate claims correctly can result in valid-but-unacceptable tokens being accepted. :contentReference[oaicite:0]{index=0}

---

### Key Establishment (ML-KEM)

This project uses **ML-KEM-768** (formerly Kyber CRYSTALS) to derive a shared secret between client and server.
[ML-KEM at nist.gov](https://csrc.nist.gov/pubs/fips/203/final)

Important:

- This is **not part of the PASETO specification**
- It is a **separate key-establishment layer**
- PASETO itself still operates as defined (v4.local symmetric encryption)

A precise description:

> PASETO is used for token format and semantics, while ML-KEM is used to establish a shared secret in a post-quantum-resistant way.

This design is intended to provide **forward-looking resistance to quantum attacks on key exchange**, not to redefine PASETO itself. :contentReference[oaicite:1]{index=1}

If you think post quantum encryption is cool, or if you believe you need it for some reason, I build this project for you and I hope you love it. ♡

### Cryptographic Primitives

This implementation uses:

- ML-KEM-768 (key encapsulation)
- XChaCha20-Poly1305 (AEAD encryption)
- Blake2b-512 (key derivation)
- PAE (Pre-Authentication Encoding)

These primitives are fixed in the implementation and are **not negotiable at runtime**, reducing algorithm confusion risks.

---

### Randomness

- Server-side: uses OS-backed CSPRNG via Rust
- Browser-side: uses `crypto.getRandomValues()`

Security depends on the correctness of the host platform's RNG.

---

### Token Confidentiality

Because this uses `v4.local`:

- Token payloads are encrypted
- Intermediaries cannot read claims without the shared secret

However:

- Footers are **not encrypted**
- Do not place sensitive data in footers

---

### Replay and Session Semantics

PASETO does not inherently prevent replay attacks.

This implementation includes:

- `jti` (token ID)
- `exp` (expiration)

Applications should consider:

- tracking used `jti` values
- binding tokens to sessions or transport context
- enforcing short TTLs

---

### Interoperability

This implementation is designed primarily for:

- OpenResty + Lua (server)
- Browser + WASM (client)

The ML-KEM integration and key derivation scheme are **specific to this project** and will probably not interoperate with other PASETO libraries.

### Standardization Status

PASETO:

- Has an Internet-Draft, but is **not a finalized IETF standard**

### Comparison to JWT

PASETO reduces certain classes of vulnerabilities by:

- eliminating algorithm negotiation
- fixing cryptographic primitives per version

However:

- It does **not eliminate the need for correct implementation**
- Security still depends on:
  - claim validation
  - key management
  - transport security
  - correct usage patterns :contentReference[oaicite:3]{index=3}

### Threat Model Notes

This design protects against:

- token tampering
- token disclosure (for v4.local)
- key exchange compromise via classical attacks

Obviously it also helps protect against

- Threat actors with quantum computers that can make practical use of [Shor's Algorithm](https://en.wikipedia.org/wiki/Shor's_algorithm).  XD
- Future threat actors with quantum computers scheming to [Harvest Now, Decrypt Later](https://en.wikipedia.org/wiki/Harvest_now%2C_decrypt_later).

It does **not** protect against:

- compromised endpoints (client or server)
- XSS exposing tokens in browser environments
- misuse of claims or authorization logic
- poor key storage practices
- your app having other bad security implementations
